# TON типы узлов

**Blockchain node** — это устройство, обычно компьютер, который запускает программное обеспечение блокчейна TON и поэтому участвует в работе блокчейна.
В целом узлы обеспечивают децентрализацию сети TON.

Узлы выполняют различные функции в рамках протокола TON:

- **Полно** и **архивные узлы** поддерживают блокчейн и историю транзакций, позволить пользователям и клиентским приложениям искать блоки и транзакции, а также отправлять новые транзакции в блокчейн;
- **Validator узлы** проверяют транзакции, обеспечивающие безопасность блокчейна.

Ниже вы найдете более подробную информацию о каждом из этих типов узлов, а также о взаимодействии полных и архивных узлов с клиентскими приложениями.

## Полный узел

**Полный узел** является основным типом узла в блокчейне TON.
Они служат основой блокчейна TON, сохраняя историю блоков - другими словами, его _текущее состояние_.

По сравнению с **архивными узлами**, полные узлы сохраняют только последнюю часть состояния блокчейна, жизненно важное значение для обеспечения стабильности сети и работы клиентских приложений.
Полные узлы _prune_ состояния блокчейна TON, хранящегося ими - это означает, что ранее блоки, которые становятся ненужными для сети, автоматически удаляются полным узлом для эффективного управления объемом данных.

Чтобы клиентские приложения могли искать блоки и транзакции, а также отправлять новые транзакции в блокчейн TON, полные узлы оснащены функциональностью liteserver: см. [Взаимодействие с узлами TON](#interacting-with-ton-nodes) ниже.

## Архивировать узел

**Архивный узел** — это полный узел, который хранит всю историю блоков блокчейна TON.
Такие узлы являются децентрализованной точкой истины с точки зрения обеспечения последовательности всей истории блокчейна.
Они служат бэкэндом для исследователей блокчейна и других приложений, основанных на глубокой истории транзакций.

Архивированные узлы не проверяют состояние блокчейна, которое повышает их системные требования, особенно с точки зрения хранения.
Согласно последним оценкам, полные узлы и узлы валидатора требуют около 1 ТБ дискового пространства, для хранения всей истории блоков в архивных узлах потребуется около 8 ТБ.

## Validator Node

**Validator Nodes** или **validators** — это участники сети TON, предлагающие новые блоки и проверяющие транзакции в них в соответствии с механизмом _Proof-Stake_ TON.
Таким образом, валидаторы вносят свой вклад в общую безопасность блокчейна.

Для успешного участия в процессе проверки валидаторы получают [награды](/v3/documentation/infra/nodes/validation/staking-incentives).

Чтобы иметь право предлагать и валидировать блоки, валидаторы избираются другими участниками по количеству удерживаемых с ними TON. их _stake_: чем больше TON разбито с валидатором, чем больше будет его шанс быть избранным, утвердить блоки для сети и получить награды.
Как правило, операторы валидаторов мотивируют других держателей TON, чтобы с ними можно было получать пассивный доход от получаемых вознаграждений.
Таким образом, валидаторы обеспечивают стабильность сети, безопасность и способствуют ее развитию.

## Взаимодействие с узлами TON

Узлы TON оснащены функцией _Liteserver_, позволяющей внешним приложениям (иными словами, _lite clients_) взаимодействовать с блокчейном TON.
Как правило, режим liteserver используется с полными и архивными узлами, в то время как узлы валидаторов не позволяют повысить производительность валидации.

Литсерверный режим позволяет клиентам отправлять транзакции через TON узлы, а также для получения информации о блоках и операциях с ними, например, для получения и обновления балансов кошельков.

У вас есть два варианта, чтобы позволить приложению lite клиента взаимодействовать с блокчейном TON:

1. Чтобы иметь стабильное соединение, вы можете запустить собственный полный или архивный узел с включённым в конфигурационный файл узла режимом Liteserver.
2. Если у вас нет возможности настроить свой собственный узел TON с Liteserver, вы можете использовать сетку публичных LiteServer, предоставленную фондом TON. Для этого используйте следующие файлы конфигурации:
   - [Общие конфигурации Liteserver - mainnet](https://ton.org/global-config.json)
   - [Конфигурация публичного сервера Liteserver - testnet](https://ton.org/testnet-global.config.json)

:::caution Использование публичных Liteservers в производстве
Из-за постоянной высокой нагрузки на публичные Liteserver, большинство из них ограничены, Именно поэтому не рекомендуется использовать их в производстве.
Это может привести к нестабильности вашего клиентского приложения.
:::
